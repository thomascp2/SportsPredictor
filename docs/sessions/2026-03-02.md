## Issue 1 — Traded Players Showing Wrong Team (Quinn Hughes VAN → MIN)

**Symptom:** Quinn Hughes (and any other recently traded player) shows their old team in the
dashboard. This was supposedly fixed on 3/1/26 but recurred on 3/2/26.

**Why yesterday's fix didn't stick:** Yesterday's fix made `sync_smart_picks()` use
`pp.get('team', '') or pred.get('team', '')` so the smart-picks row would have the correct
team. But the fix was incomplete — `sync_predictions()` runs **before** `sync_smart_picks()`
and writes ALL raw predictions to Supabase using the local DB team (stale). Then when
`sync_smart_picks()` called `SmartPickSelector.get_smart_picks()`, it hit this check:

```python
if pp_team and pred_team and pp_team != pred_team:
    continue  # ← SKIPPED the player entirely
```

Because PP says `team='MIN'` but the local prediction says `team='VAN'`, the smart pick was
skipped and never ran. Nobody overwrote the stale VAN row that `sync_predictions()` had
already written. The row stayed VAN.

**Root cause (architectural):** Two sync operations write to the same Supabase rows. The
first (`sync_predictions`) uses local DB team; the second (`sync_smart_picks`) was supposed
to correct it but skipped traded players entirely.

**Fix — two-pronged (permanent):**

1. **`SmartPickSelector.get_smart_picks()`** (`shared/smart_pick_selector.py` line 333):
   Changed `continue` to a log message. Traded players are no longer skipped — PP team is
   already used as authoritative at SmartPick creation via `pp.get('team', '') or pred.get('team', '')`.
   Probability calculation is valid regardless of team (it's based on player stats, not opponent).

   ```python
   # Before:
   if pp_team and pred_team and pp_team != pred_team:
       continue  # silently dropped the player

   # After:
   if pp_team and pred_team and pp_team != pred_team:
       print(f"[INFO] Trade detected: {pp['player_name']} local={pred_team} → PP={pp_team}")
       # continue removed — SmartPick creation uses pp.get('team') as authoritative
   ```

2. **`sync_predictions()`** (`sync/supabase_sync.py`): Before building the props list, loads
   today's PP lines from `prizepicks_lines.db` and builds a `player_name_lower → current_team`
   lookup. Any prediction whose local team differs from PP's team gets corrected **before**
   the Supabase upsert — so even the base prediction row starts with the right team.

   ```python
   pp_team_lookup = {}  # built from prizepicks_lines WHERE start_time date = game_date
   # In the row loop:
   pp_team = pp_team_lookup.get(norm_name, '')
   if pp_team and local_team and pp_team.upper() != local_team.upper():
       print(f"[SYNC] Trade correction: {player} {local_team} → {pp_team}")
       team = pp_team
   else:
       team = local_team
   ```

## Issue 2 — Multiple Standard Lines per Player-Prop

**Symptom:** Jokic shows four standard lines for assists (5.5 STD, 7.5 STD, 8.5 STD, 9.5 STD).
Platform rule: max 1 standard line per (player, prop). Bub Carrington also showed 2.5 STD
and 3.5 STD for assists simultaneously.

**Root cause:** PrizePicks API returns multiple projections with `odds_type='standard'` for
the same player+prop — each with a unique `projection_id`. The database schema uses
`UNIQUE(projection_id, fetch_date)` as its unique key, so all four rows persist.
`_get_pp_lines()` queries `SELECT DISTINCT player_name, prop_type, line, odds_type, team` —
DISTINCT on all 5 columns — so all four rows survive (each has a different `line` value).

**Fix:** After the DB query in `_get_pp_lines()` (`shared/smart_pick_selector.py` lines 477+),
deduplicate standard lines per (player, prop) before returning. Keeps the **median** standard
line value, which is most likely the real board line (outliers at either extreme are de-facto
goblin/demon lines mislabeled by the API). Logs when it culls extras.

```python
# Enforce platform rule: max 1 standard line per (player, prop)
std_by_key = defaultdict(list)
for r in all_rows:
    if r['odds_type'] == 'standard':
        std_by_key[(r['player_name'], r['prop_type'])].append(r)

seen_std = set()
deduped = []
for r in all_rows:
    if r['odds_type'] != 'standard':
        deduped.append(r)
        continue
    key = (r['player_name'], r['prop_type'])
    if key in seen_std:
        continue
    candidates = sorted(std_by_key[key], key=lambda x: x['line'])
    keeper = candidates[len(candidates) // 2]   # median
    deduped.append(keeper)
    seen_std.add(key)
return deduped
```

## Issue 3 — Probability Inversion (Higher Line = Higher Probability)

**Symptom:** Jokic 7.5 STD assists shows 60%, but 8.5 STD shows 65%. Mathematically
impossible for the same player with the same distribution — OVER 8.5 must be ≤ OVER 7.5.

**Root cause:** When multiple prediction rows exist for the same (player, prop) — which
happens when duplicate standard lines cause multiple prediction rows to be generated — the
smart pick selector's `pred_lookup[key]` list is populated in arbitrary DB row order.
`pred_lookup[key][0]` picks the first row, but different PP lines may map to different
prediction rows (with slightly different `f_l10_avg` values from different feature
extraction runs), producing non-monotonic probabilities.

**Fix:** Sort `pred_lookup[key]` by probability descending after building the lookup, so
`[0]` always uses the most confident/most-data-supported prediction. Combined with the
standard line deduplication fix (Issue 2), this eliminates the primary source of duplicates.

```python
for key in pred_lookup:
    pred_lookup[key].sort(key=lambda p: p.get('probability', 0), reverse=True)
```

## Issue 4 — Game Times Disappearing Every Day

**Symptom:** Dashboard `Time` column was blank on 3/2/26. It worked on 3/1/26 only.

**Root cause:** `sync_odds_types()` and `sync_game_times()` were **never wired into the
orchestrator**. They were run manually during the 3/1/26 session as a one-time fix and
never called again. The orchestrator only called `sync_predictions()` and `sync_smart_picks()`.

**Fix:** Added both to the orchestrator's prediction pipeline sync block
(`orchestrator.py` lines 514+):

```python
# After sync_predictions + sync_smart_picks:
odds_sync = syncer.sync_odds_types(self.config.sport.lower(), target_date)
time_sync = syncer.sync_game_times(self.config.sport.lower(), target_date)
```

Both now run automatically after every prediction run. Will never need manual re-running.

## Bug — Goblin/Demon UNDERs Appearing in Dashboard

**Symptom:** Kobe Brown IND vs MEM showing `UNDER rebounds 6.5 GOB` in Today's Picks.

**Root cause:** `sync_odds_types()` only corrects rows that were touched by smart picks.
If a player's pick is below break-even for a goblin line (76%), smart picks skips it and
`sync_odds_types()` never flips its `ai_prediction` from UNDER to OVER.

**Fix:** Added client-side post-filter in `fetch_picks()` in `cloud_dashboard.py`:
```python
df = df[~(df["odds_type"].isin(["goblin", "demon"]) & (df["ai_prediction"] == "UNDER"))]
```
