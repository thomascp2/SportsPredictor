## Pick Pool Narrowing Recommendations

### Root Cause: Why 200+ T1 Picks?
With 1,300+ NBA predictions/day across 14 prop types × many lines × ~100 players, every player gets modeled on every available line. Even at T1 threshold, this generates 200+ picks because:
1. Models are highly accurate (69–99%) → most predictions land at 75%+
2. No per-player deduplication (player gets T1 on points, rebounds, assists, pra all at once)
3. No daily cap on how many picks surface

### Recommended Filters (implement in SmartPickSelector or Supabase query):

#### NBA
1. **Standard lines only** for the main "picks of the day" view (goblin/demon in separate tab)
2. **Per-player max 2 picks**: highest-edge point prediction + highest-edge combo stat (pra/pts_rebs/pts_asts)
3. **UNDER bias**: auto-weight UNDER picks higher — they hit 79-93% vs OVER at 58-72%
4. **Disable threes OVER**: zero hit rate — pure noise. Add guard: `if prop_type == 'threes' and prediction == 'OVER': skip`
5. **Daily cap**: Top 30 picks by edge → gives manageable, actionable list
6. **Prop focus**: Prioritize points + rebounds OVER, pra UNDER, points UNDER (high lines) — clearest edges

#### NHL
1. **T1-T3 only**: T4 (56%) and T5 (51.7%) are at or below break-even — no edge
2. **UNDER priority**: points 1.5 UNDER (88.2%) and shots 3.5 UNDER (88.3%) are monster edges
3. **Avoid shots 3.5 OVER**: 40.7% — actively harmful
4. **Per-player max 1 pick**: highest edge pick only

### Parlay Strategy
Given 200 T1 picks — betting 200 singles is not practical. Instead:
- **3-5 leg parlays of T1 standard picks only**: 71-85% hit rate per leg
- **UNDERs-only parlay**: pra UNDERs, points UNDERs at high lines, rebounds UNDERs at high lines — expected value ~85%^N per leg
- **Target**: 3-leg parlay of T1 UNDERs ≈ 85% × 85% × 85% = 61% hit rate at 6x payout = +EV

## MLB Guidance

### Current State
- Prework done in a separate branch
- NHL/NBA lessons learned applicable

### Architecture Recommendations

**What transfers directly from NHL/NBA:**
- Orchestrator pattern (sport config, scheduling, error handling)
- SQLite database design (mirror NHL schema — JSON features blob)
- SmartPickSelector (edge-based tier logic)
- API self-healing system
- Supabase sync layer
- Discord bot commands (add `!picks mlb`, `!refresh mlb`)

**MLB-specific considerations:**

1. **Data sources**: MLB Stats API (`statsapi.mlb.com`) is free and reliable. ESPN API also covers MLB. No API key needed.

2. **Prop types to target first**:
   - `hits` O0.5/1.5 (most liquid, low variance)
   - `total_bases` O1.5/2.5 (good volume)
   - `strikeouts_pitcher` O4.5/5.5/6.5 (high edge potential, pitcher is primary factor)
   - `home_runs` O0.5 (goblin-style, popular)
   - Start with 3-4 props, not all at once

3. **Key features for MLB** (different from basketball):
   - Pitcher vs batter handedness matchup (CRITICAL)
   - Ballpark factors (home run friendly vs pitcher friendly)
   - Pitcher ERA/WHIP last 3 starts
   - Batter vs LHP/RHP splits
   - Recent form (L10 games)
   - Weather/wind for home run props

4. **Season timing**: MLB season starts ~March 27, 2026. You have ~12 days.
   - Week 1: Set up database schema + API connectors
   - Week 2: Prediction pipeline + grading pipeline
   - Learning mode for 4-6 weeks before ML training

5. **Gotchas unique to MLB**:
   - Double-headers (2 games same day, same players) — handle date+game_id not just date
   - Pitcher scratches happen day-of — must re-check lineup at game time
   - Stats API uses `gamePk` not `gameId` — different naming
   - Lineup confirmation not until ~3-4 hours before game — predictions should be generated after lineup lock

6. **Copy base**: Start from `nhl/` directory structure (JSON features) not `nba/` (individual columns). NHL's rolling stats approach works better for baseball's smaller sample sizes.
