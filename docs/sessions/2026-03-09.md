## 1. ML Retrain Investigation — Why Sunday (Mar 8) Didn't Fire

### Finding

The `schedule` library bakes task times at process startup. The orchestrator starts in the middle of
the night (triggered by NHL grading at 3AM, NBA grading at 5AM). Retrain was scheduled at 2:00AM /
2:30AM — **before** the orchestrator process was running. DST shift on Mar 8 likely moved the
effective wall clock window further, making the miss worse.

### Fix (`orchestrator.py`)

| Task | Old Time | New Time |
|------|----------|----------|
| NHL retrain | 02:00 CST | 03:30 CST |
| NBA retrain | 02:30 CST | 05:30 CST |

Both now fire **after** the grading runs that guarantee the orchestrator is alive. Only the
retrain schedule change requires an orchestrator restart (bat file); all other changes hot-load.

## 2. Probability Calibration Architecture Fix (`ml_training/train_models.py`)

### Problem

Previous 3-way split: 70% train / 15% val / 15% test.
Isotonic calibration was fit on the **val set** — the same set used for model selection. This
causes data leakage: the calibrator sees exactly the distribution that chose the model, biasing
confidence estimates upward (especially on OVERs, ~+10% overconfidence gap discovered).

### Fix — 4-Way Split

| Split | Size | Purpose |
|-------|------|---------|
| train | 60% | Model training |
| val | 15% | Model selection (hyperparameter tuning) |
| cal | 10% | Isotonic calibration (dedicated, no leakage) |
| test | 15% | Final evaluation |

`MLConfig` gains `cal_size: float = 0.10`. `prepare_data()` now returns 4 splits.
`calibrate_best_model()` is a new method that runs only on the winning model against the clean
cal set after selection is complete.

## 3. Goblin / Demon / Standard Tier Fix (`shared/smart_pick_selector.py`)

### Problem

`_get_tier()` used raw probability thresholds regardless of odds type:

```python
# OLD (wrong)
if probability >= 0.75: return 'T1-ELITE'
```

A goblin pick at 75% probability is a **losing bet** — break-even is 76%. It was being labeled
T1-ELITE and sent to users.

### Break-Even Math

| Odds Type | Break-Even | Notes |
|-----------|-----------|-------|
| standard | 0.56 (56%) | Normal PrizePicks line |
| goblin | 0.76 (76%) | Easier line, lower payout |
| demon | 0.45 (45%) | Harder line, higher payout |

### Fix — Edge-Based Tiers

`ai_edge = (ai_probability − break_even) × 100` is computed on every row during sync.
Tiers now use edge instead of raw probability:

```python
def _get_tier(self) -> str:
    edge = self.edge  # already computed as (prob - break_even) * 100
    if edge >= 19: return 'T1-ELITE'
    elif edge >= 14: return 'T2-STRONG'
    elif edge >= 9: return 'T3-GOOD'
    elif edge >= 0: return 'T4-LEAN'
    else: return 'T5-FADE'
```

Filter also changed: picks are now filtered by edge >= 0 (profitable), not probability >= min_prob,
so demon picks at 52% (well above 45% break-even) aren't wrongly rejected.

## 6. Quinn Hughes Trade Bug (V6) — Resurfaced

### What happened

Quinn Hughes (VAN → MIN, traded months ago) appeared on VAN in today's app picks. Same class of
bug as the NBA team assignment fix (Mar 3) — historical logs from old team polluting current roster.

### Root Cause

`nhl/scripts/generate_predictions_daily_V6.py::get_players_with_history_for_team()` had NO
`game_date` parameter and queried ALL historical logs without ROW_NUMBER filtering. A traded player
like Hughes has years of VAN game logs and still shows up on the VAN roster query.

The orchestrator uses **V6** (not V5). The ROW_NUMBER CTE fix applied to V5 on Mar 3 was never
ported to V6.

### Fix Applied

Rewrote `get_players_with_history_for_team()` in V6 with ROW_NUMBER CTE:

```python
def get_players_with_history_for_team(team: str, game_date: str, ...) -> list[str]:
    cursor.execute('''
        WITH current_teams AS (
            SELECT player_name, team AS current_team
            FROM (
                SELECT player_name, team,
                       ROW_NUMBER() OVER (PARTITION BY player_name ORDER BY game_date DESC) AS rn
                FROM player_game_logs
                WHERE game_date < ?
            ) ranked
            WHERE rn = 1
        )
        SELECT pgl.player_name, COUNT(*) AS games,
               SUM(pgl.points) * 1.0 / COUNT(*) AS ppg
        FROM player_game_logs pgl
        JOIN current_teams ct ON pgl.player_name = ct.player_name
            AND ct.current_team = ?
        WHERE pgl.team = ?
          AND pgl.game_date < ?
        GROUP BY pgl.player_name
        HAVING games >= ?
        ORDER BY ppg DESC
        LIMIT ?
    ''', (game_date, team, team, game_date, min_games, top_n))
```

Updated call site (line 613) to pass `game_date`:
```python
players = get_players_with_history_for_team(team, game_date, min_games=5, top_n=players_per_team)
```
