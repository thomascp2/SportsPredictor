## Issue 1 — Kevin Durant (and 40+ others) Predicted for the Wrong Team

This is the main issue of the session. Here is the full root-cause chain.

### Background: How predictions are generated

Every day the prediction script (`generate_predictions_daily_V6.py`) does the following:

1. Fetch today's schedule to find which teams are playing
2. For each home/away team pair, call `_get_team_players()` to get the roster
3. Match those roster players against active PrizePicks lines
4. Generate a prediction for each matched (player, prop, line) combo

Step 2 is where the bug lived.

### Root cause: `_get_team_players()` used all-time history, not current team

The original query looked like this:

```sql
SELECT player_name, AVG(minutes) as avg_minutes, COUNT(*) as games
FROM player_game_logs
WHERE team = ?
  AND game_date < ?
GROUP BY player_name
HAVING games >= 3
ORDER BY avg_minutes DESC
LIMIT ?
```

The parameter `?` was the team abbreviation for today's game — say, `HOU` for Houston.

**The bug:** Kevin Durant was traded from Phoenix (PHX) to Houston (HOU) in early 2025.
The `player_game_logs` table contains every game he has ever logged — including all of his
PHX games since October 2024. Because the query filters `WHERE team = ?` with no date
constraint, it pulls every row where `team = 'PHX'`, which includes KD's entire PHX history.
So when the script runs for a PHX game, KD shows up in Phoenix's roster — even though
he has been a Rocket for months.

The same is true for every trade that season: KD PHX→HOU, LaVine CHI→SAC, Fox SAC→SA,
and 37+ others. Each traded player's old game logs still carry the old team abbreviation,
so they permanently "haunt" their former team's roster in the prediction script.

The result: when PHX played a game, the script would predict KD props as if he were still
a Sun. PHX is his "roster" per the historical logs. The model would also assign him the
wrong `is_home` flag (whether Phoenix is home, not whether Houston is home) and the wrong
opponent.

### Compounding bug: ESPN abbreviation mismatches

Six teams use different abbreviations in the ESPN schedule API vs. the NBA Stats API
(which populates `player_game_logs`):

| ESPN (schedule) | NBA Stats API (logs) |
|-----------------|----------------------|
| WSH | WAS |
| NY  | NYK |
| SA  | SAS |
| NO  | NOP |
| GS  | GSW |
| UTAH | UTA |

When the prediction script received ESPN's schedule and got `team = 'WSH'`, it searched
`player_game_logs WHERE team = 'WSH'` — but all Washington logs are stored as `'WAS'`.
The result: zero players returned for Washington, so no predictions were generated for any
Washington player. Same for the other five teams.

### The fix: CTE with ROW_NUMBER() + TEAM_ALIASES

**Part 1 — "Most recent team" CTE:**

Instead of querying all logs for a given team, we first determine each player's *current*
team by finding their most recent game log entry:

```sql
WITH current_teams AS (
    SELECT player_name, team AS current_team
    FROM (
        SELECT player_name, team,
               ROW_NUMBER() OVER (PARTITION BY player_name ORDER BY game_date DESC) AS rn
        FROM player_game_logs
        WHERE game_date < ?
    ) ranked
    WHERE rn = 1
)
SELECT pgl.player_name, AVG(pgl.minutes) AS avg_minutes, COUNT(*) AS games
FROM player_game_logs pgl
JOIN current_teams ct ON pgl.player_name = ct.player_name
    AND ct.current_team IN (?, ?)
WHERE pgl.team IN (?, ?)
  AND pgl.game_date < ?
GROUP BY pgl.player_name
HAVING games >= 3
ORDER BY avg_minutes DESC
LIMIT ?
```

The `current_teams` CTE finds each player's team as of their last logged game before today.
The main query then requires that this current team matches the team we're looking up. KD's
most recent log entry is `team='HOU'`, so he no longer appears when we query PHX's roster.

**Part 2 — TEAM_ALIASES dict:**

```python
TEAM_ALIASES = {
    'WSH': 'WAS', 'WAS': 'WSH',
    'NY':  'NYK', 'NYK': 'NY',
    'SA':  'SAS', 'SAS': 'SA',
    'NO':  'NOP', 'NOP': 'NO',
    'GS':  'GSW', 'GSW': 'GS',
    'UTAH': 'UTA', 'UTA': 'UTAH',
}
```

For any team abbreviation, we compute its alias and pass both variants as parameters to
the `IN (?, ?)` clauses above. This ensures a WSH schedule entry correctly retrieves
WAS logs, and vice versa.

**Part 3 — Removed FALLBACK_PROPS:**

Previously, when a player had no PrizePicks line match, the script fell back to generating
predictions at fixed lines (`FALLBACK_PROPS = {'points': [15.5, 20.5, 25.5], ...}`). This
created "phantom" predictions for lines that weren't actually on the board. Removed this
fallback: if no PP line matches, skip the player. Cleaner training data.

### Files changed

- `nba/scripts/generate_predictions_daily.py` — TEAM_ALIASES + CTE

streaks) were all correct — the grader finds players by NAME, not team, so outcome labels
are correct.

NHL models do use 5 opponent features each, but only 8 NHL players were on wrong teams,
an even smaller fraction of training data.

The contamination will naturally dilute as clean data accumulates post-fix. The next
scheduled retrain (first Sunday with 500+ new predictions) will incorporate the cleaner data.

## Issue 2 — Prediction Script Hanging (Windows Defender)

After applying the code fixes, the prediction script hung with zero output for 5+ minutes.
Diagnosis traced the hang to `import scipy.stats` and `import sklearn` — both hung
indefinitely at the OS level regardless of timeout, thread env vars, or directory.

**Root cause:** Windows Defender performs real-time scanning of `.pyd` (compiled C extension)
files. When multiple Python processes were killed during the session, their file handles on
scipy/sklearn `.pyd` files were released, causing Windows Defender to re-scan them on the
next import attempt. This can block indefinitely on Windows with the default Defender config.

**This was the same scipy that worked at 12:01 PM** — the code was correct, but the system
state changed because of the process kills earlier in the session.

**Fix applied:** Moved the `from scipy.stats import poisson, norm` in `statistical_predictions.py`
from module level into a lazy `_get_scipy()` helper:

```python
_poisson = None
_norm = None

def _get_scipy():
    global _poisson, _norm
    if _poisson is None:
        from scipy.stats import poisson as _p, norm as _n
        _poisson, _norm = _p, _n
    return _poisson, _norm
```

`norm.cdf()` in `predict_continuous_prop()` now calls `_get_scipy()[1].cdf(...)`. This means
scipy is only imported when the statistical predictor actually runs a continuous prediction —
never at module load time. Since LEARNING_MODE is False and ML models handle most predictions,
scipy may not be loaded at all on a typical run.

The underlying Windows issue was not resolved — the permanent fix is to add Python's
`site-packages` directory to Windows Defender exclusions in the Windows Security settings.

**Note:** Because games were already in progress by the time the hang was resolved, tonight's
predictions were not regenerated. The Mar 3 gap is minor (one game day) and has negligible
effect on training data.

## Feature 1 — Automated Weekly ML Retraining

Added to `orchestrator.py`:

**Schedule:** Every Sunday — NHL at 2:00 AM CST, NBA at 2:30 AM CST (staggered; both before
their respective grading runs). Configured via `self.retrain_time` on `SportConfig`.

**Condition gate:** Only runs if 500+ new predictions exist in the sport's SQLite DB since
the last model's `trained_at` date. This prevents wasteful retrains during slow weeks
(e.g. playoffs with few games, or after a vacation gap).

**Implementation:**
- `_get_last_train_date()` — reads `trained_at` from the newest model's `metadata.json` in
  the model registry
- `_count_new_predictions_since(date_str)` — queries `SELECT COUNT(*) FROM predictions WHERE game_date > ?`
- `run_weekly_ml_retrain()` — checks condition, spawns `train_models.py --sport {sport} --all`
  as a subprocess (30 min timeout), calls `_send_weekly_retrain_notification()`
- `_send_weekly_retrain_notification()` — reads newly-written metadata files to compute
  accuracy deltas vs. previous version; posts compact Discord message

**Discord notification format (on success):**
```
[NBA] ML RETRAIN — Sun Mar 8

14 models updated | 847 new preds since Mar 2 | avg +2.1% acc

Top gains:
  points_O25.5: 83.1% (+3.2%)
  assists_O7.5: 87.5% (+2.1%)
  threes_O2.5: 99.1% (+0.8%)

Degraded:
  rebounds_O10.5: 71.2% (-1.8%)
```

**On failure:** Short alert with log file path. Old models remain active — the registry is
never modified if the training subprocess fails.
