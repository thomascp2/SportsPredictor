## Outstanding Issues — Specific Instructions for Next Agent

### Issue 1: Line Duplication Inflation (HIGHEST PRIORITY)

**What it is:** The V6 prediction generator creates one prediction row per PrizePicks line per player/prop, not one per player/prop. A player with 10 lines (e.g., Paolo Banchero points at 17.5, 19.5, 21.5...34.5) gets 10 prediction rows for the same game. Grading scores all 10 against the same actual value. If he scores 28, all lines below 28 hit → 7 HITs from 1 performance.

**Evidence (NBA, last 30 days):**
```
Andrew Wiggins pts_rebs 2026-04-14: 10 lines (14.5-29.5)
Paolo Banchero points 2026-04-15:   10 lines (17.5-34.5)
Paul George pra 2026-04-15:         10 lines (24.5-44.5)
```

**Inflation factors per prop (unique player-games vs raw rows):**
```
pra UNDER:     11,089 raw / 4,595 unique = 2.41x inflation
points UNDER:   9,911 raw / 4,731 unique = 2.09x inflation
points OVER:    6,421 raw / 3,354 unique = 1.91x inflation
```

**Why some props look impossibly good:**
- `NHL points OVER`: 99.0% — model always picks OVER at the lowest line in the stack; all low lines hit
- `NBA blocked_shots OVER`: 95.3% — same artifact
- `NBA steals OVER`: 97.7% — same
- `MLB batter_strikeouts OVER`: 92.2% — same
- `MLB earned_runs OVER`: 89.7% — same

These are not real edges. They're artifacts of grading every line in the stack.

**How to fix — two parts:**

**Part A: Fix accuracy reporting (lower risk, do first)**
In `tools/db_cleanup.py` (or a new `tools/dedup_outcomes.py`), add a function that identifies duplicates and keeps only the prediction row whose `line` is closest to the actual `standard` PP line (i.e., the canonical line, not the goblin/demon variants). For smart picks, PrizePicks only offers ONE line per player/prop — we should only count that one.

The de-dup key is: `(player_name, prop_type, game_date, prediction)` — keep the row where `line` is closest to the standard PP line for that player/prop/date. Tag the others with `data_quality_flag = 'DUPLICATE_LINE'`.

Query to find dupes:
```sql
SELECT player_name, prop_type, game_date, prediction, COUNT(*) as n
FROM prediction_outcomes
WHERE is_smart_pick = 1 AND outcome IN ('HIT','MISS')
GROUP BY player_name, prop_type, game_date, prediction
HAVING COUNT(*) > 1
```

**Part B: Fix the prediction generator (prevents future duplication)**
In `nba/scripts/generate_predictions_daily_V6.py` and `nhl/scripts/generate_predictions_daily_V6.py`, find where predictions are inserted and add a dedup step: for a given `(player_name, prop_type, game_date)`, only generate one prediction — at the `standard` line if available, else at the closest-to-median available line. This is the canonical prediction that gets graded.

The goblin/demon line predictions should still be generated for pp-sync (SmartPickSelector needs them for edge calculation), but should be stored with a flag that prevents them from being double-counted in grading.

### Issue 3: MLB UNDER Props Systematically Miscalibrated

Beyond the prop-level guards above, MLB UNDER predictions broadly underperform. The model generates UNDER probabilities using statistical distributions fit to historical averages, but MLB pitcher/batter prop lines are set asymmetrically — PP tends to set lines that favor OVER for pitchers (strikeouts, outs_recorded) and UNDER loses systematically.

**Next investigation:** After adding the guards above, re-run the breakdown query and check if the remaining MLB UNDER combos (hits_allowed UNDER, doubles UNDER, rbis UNDER, etc.) are above or below break-even. There may be additional guards needed.

```python
# Run this query in mlb/database/mlb_predictions.db
SELECT prop_type, prediction, COUNT(*), 
       ROUND(100.0*SUM(CASE WHEN outcome='HIT' THEN 1 ELSE 0 END)/COUNT(*),1) AS pct
FROM prediction_outcomes
WHERE is_smart_pick=1 AND outcome IN ('HIT','MISS')
  AND data_quality_flag IS NULL
GROUP BY prop_type, prediction
HAVING COUNT(*) >= 10
ORDER BY pct ASC
```
