### 2. db_cleanup.py Performance Fix (commit 320ba3cd)

The original `_fix_odds_type_labels()` and `_fix_line_duplication()` used `ATTACH DATABASE`
+ cross-join with `LOWER()` wrapping against 926K rows — defeated all indexes, hung indefinitely.

Rewrote both functions to load prizepicks_lines into Python dicts (one bulk read per sport),
then do O(1) dict lookups. NHL now completes in seconds; all three sports in under a minute.

Also fixed a Unicode `→` arrow in a label string causing cp1252 crash on Windows.

### 3. Generator odds_type Fix (commit 320ba3cd)

Root cause of all historical mislabeling: V6 generators fetched all odds_types but lost the
odds_type per line. Every prediction was stored with odds_type=NULL.

**NBA fix:** `get_all_pp_player_lines()` now returns `(line, odds_type)` tuples. Generation
loop unpacks both. `_save_prediction()` accepts `odds_type` and stores it in the INSERT.

**NHL fix:** `get_all_pp_player_lines()` returns `(player_lines, odds_type_map)`. After all
predictions for the day are saved, a backfill UPDATE corrects predictions.odds_type using
the map. (NHL's `_save_prediction` is in the statistical engine — threading through it would
be invasive.)

**Why this matters:** grading runs at 3 AM, before pp-sync at 12:30 PM. prediction_outcomes
inherits odds_type from predictions at grading time. With the fix, odds_type is correct before
grading ever runs.

### 4. Key Finding: smart_pick_selector Already Correct

The selector already uses odds_type-specific break-evens (`compute_break_even()`), sigma gates
for goblin picks, and has a permanent demon OVER guard. It reads odds_type from prizepicks_lines
directly (not from predictions), so selection quality was never affected. The bug was storage-only.
