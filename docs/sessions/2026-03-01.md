## Issue 1 — NHL Duplicate Player Rows (S. Bennett vs Sam Bennett)

**Symptom:** Both `S. Bennett` (prediction row) and `Sam Bennett` (smart-picks row) existed for the same player, same date, same prop.

**Root cause:** Two separate sync operations write to `daily_props` with different canonical names:
- `sync_predictions()` uses the local NHL DB name → `"S. Bennett"` (abbreviated)
- `sync_smart_picks()` uses `pick.player_name` which comes from PrizePicks → `"Sam Bennett"` (full)

The Supabase upsert conflict key is `(game_date, player_name, prop_type, line)` — exact string match.
`"S. Bennett"` ≠ `"Sam Bennett"`, so instead of updating the prediction row the smart-picks sync
**inserted a second row**. The prediction row stayed `odds_type='standard'`, never updated to goblin.

**Fix:** Added `local_player_name: str` field to the `SmartPick` dataclass in `smart_pick_selector.py`.
This field is populated from `pred['player_name']` (the local DB abbreviated name) at pick creation time.
`sync_smart_picks()` now uses `pick.local_player_name` as the canonical key for the upsert, matching
the name already in the prediction row.

```python
# SmartPick dataclass (smart_pick_selector.py)
@dataclass
class SmartPick:
    player_name: str         # PrizePicks display name ("Sam Bennett")
    local_player_name: str   # Local DB name, may be abbreviated ("S. Bennett")
    team: str
    ...

# In get_smart_picks(), when creating the pick:
pick = SmartPick(
    player_name=pp['player_name'],
    local_player_name=pred['player_name'],   # ← NEW: carries local DB name
    team=pp.get('team', '') or pred.get('team', ''),
    ...
)

# In sync_smart_picks():
canonical_name = self._normalize_name(
    pick.local_player_name if pick.local_player_name else pick.player_name
)
```

**Cleanup:** Manually deleted 37 orphan full-name rows from Supabase for 2026-03-01, then re-ran
smart picks sync. 37/37 synced correctly as updates to existing prediction rows.

## Issue 2 — NHL/NBA odds_type Stuck at "standard" for Goblin Lines

**Symptom:** Goblin lines (SOG 1.5 for Tkachuk/Marchand, SGA steals 0.5, Hartenstein points 7.5)
all showing `odds_type = 'standard'` in the dashboard.

**Root cause:** `sync_smart_picks()` recalculates edge using a Normal distribution model, then filters
with `min_edge=0`. For goblin lines, the break-even probability is 76% (vs 56.5% for standard).
The model may compute, e.g., SGA steals 0.5: `Normal(mean=0.9, std=0.567) → P(x>0.5) ≈ 75.97%`.
Edge = 75.97% - 76% = **-0.03%** → filtered out by min_edge=0. The pick is never included in the
smart-picks upsert, so the prediction row's `odds_type` field stays as whatever `sync_predictions()`
wrote initially (`'standard'`).

The underlying issue: `odds_type` is a **factual label** from PrizePicks, not a derived quality metric.
We should never compute it from our model's edge calculation.

**Fix:** New `sync_odds_types()` method in `supabase_sync.py`:
- Reads all rows from `daily_props` for the target date (with Supabase pagination — see Bug #1 below)
- For each row, looks up the PP DB to get the actual `odds_type` for that player/prop/line
- For NHL abbreviated names, uses `_initial_match()` to bridge `"S. Bennett"` → `"Sam Bennett"`
- Updates `odds_type` directly in Supabase — **no edge filtering**, no model involved
- Runs automatically after `sync_smart_picks()` in the pipeline

```python
def sync_odds_types(self, game_date: str, sport: str) -> dict:
    """Sync odds_type from PrizePicks DB to Supabase — no edge filtering."""
    # Pull all Supabase rows (paginated)
    all_rows = []
    page_size = 1000
    offset = 0
    while True:
        r = (self.client.table('daily_props')
             .select('id,player_name,prop_type,line')
             .eq('game_date', game_date).eq('sport', sport.upper())
             .range(offset, offset + page_size - 1).execute())
        batch = r.data or []
        all_rows.extend(batch)
        if len(batch) < page_size:
            break
        offset += page_size

    # For each row, find odds_type in PP DB and update
    for row in all_rows:
        pp_type = self._lookup_pp_odds_type(row, game_date, sport, conn)
        if pp_type and pp_type != row.get('odds_type'):
            self.client.table('daily_props').update(
                {'odds_type': pp_type}
            ).eq('id', row['id']).execute()
            corrections += 1
```

## Issue 3 — NBA Team Staleness (Traded Players)

**Symptom:** De'Aaron Fox listed as SAC Kings in the dashboard. Nine other traded NBA players also on wrong teams (Lopez, Dosunmu, Martin, Thomas, Johnson, LeVert, Yabusele, Achiuwa).

**Root cause:** The NBA V6 prediction generator assigns team by querying `player_game_logs` for the
player's most recent game entry. For traded players, the most recent logs still show the old team
until new games accumulate. PrizePicks, by contrast, always reflects the current roster.

**Fix:** In `SmartPick` creation, PP team is now authoritative:
```python
team=pp.get('team', '') or pred.get('team', ''),
```
PP team is tried first; local DB team is only used as fallback if PP has no team. Since
`sync_smart_picks()` upserts (not inserts) into existing prediction rows, the team field gets
corrected in Supabase as soon as a pick is found in PP — usually the same day.

**Cleanup:** 9 team mismatches manually corrected in Supabase for 2026-03-01 data.

## Issue 4 — Game Time Missing from Dashboard

**Request:** User wanted tip-off time displayed per pick so they know how much time they have
to build parlays between pick generation (6:15 AM CST) and game start.

**Implementation:** New `sync_game_times()` method in `supabase_sync.py`:
- Reads `start_time` from `prizepicks_lines.db`, grouped by team for each sport/date
- Stores raw ISO TIMESTAMPTZ (e.g. `"2026-03-01T13:10:00-05:00"`) in `daily_props.game_time`
- Runs in pipeline after `sync_odds_types()`

**Dashboard update:** New `Time` column in picks table, formatted via `_fmt_time()`:
- Parses ISO offset from the stored timestamp
- Converts to ET (UTC-5 in winter)
- Displays as `"1:10 PM ET"`, `"7:30 PM ET"`, etc.

**Bugs encountered during implementation:**
1. **Missing GROUP BY**: First version of `sync_game_times()` returned only 1 row (global MIN across all teams). Fixed by adding `GROUP BY team` to the SQL query.
2. **TIMESTAMPTZ mismatch**: First attempt stored `"1:10 PM ET"` (human-readable string), but Supabase `game_time` column is `TIMESTAMPTZ`. Error: `invalid input syntax for type timestamp with time zone`. Fixed by storing raw ISO: `"2026-03-01T13:10:00-05:00"`.

**Result:** 3,068 NBA + 266 NHL game time rows populated.

## Bug #1 — Supabase 1000-Row Pagination Cap

**Symptom:** First run of `sync_odds_types()` corrected only 352 rows. SGA and Hartenstein still
showing `standard` despite the fix being deployed.

**Root cause:** Supabase `select()` returns at most 1000 rows by default. NBA `daily_props` had
2,800+ rows. The first 1000 were corrected; the rest (including SGA, Hartenstein) were never
fetched.

**Fix:** Added pagination loop using `.range(offset, offset + page_size - 1)` to both
`sync_odds_types()` and `sync_game_times()`. After the fix, re-ran and corrected all 905 rows.
