## 3. Grading Partial-Round Score Bug

**Problem**: We manually ran `auto_grade_daily.py 2026-03-29` while Round 4 of the Houston Open was still in progress. ESPN's leaderboard API returns a live cumulative hole-by-hole score (e.g. `35` for a player through 9 holes) as a real non-None value. The grading script checked only `score is None`, so partial scores like 35 were treated as final round scores. 121 out of 212 outcomes were graded incorrectly.

**Example**: Denny McCarthy showed `actual_value = 35` and was graded HIT on UNDER 72.5 while still on the course.

**Investigation**:
- Called ESPN API live to check status values
- In-progress players have `status = "STATUS_IN_PROGRESS"`
- Completed players: `STATUS_FINISH` or `STATUS_CUT`

**First attempt (rejected)**: Guard `if actual_value < 60: skip`. The user correctly flagged this could block a legitimate 59 (PGA Tour record rounds include 58 by Jim Furyk, 59 by multiple players).

**Final fix** (`golf/scripts/auto_grade_daily.py`):
```python
player_status = entry.get("status", "")
if player_status == "STATUS_IN_PROGRESS" and round_num == current_round:
    skipped += 1
    continue
```

**Key nuance**: The `round_num == current_round` condition is critical. A STATUS_IN_PROGRESS player's *past* rounds (R1, R2, R3) are already final and should be graded. Only their *current active round* has a partial score. Same guard added to `also_store_round_logs()` to prevent partial scores entering the training data.

**Data remediation**: Deleted 121 bad outcomes (scores 19–58). Re-ran grading — 0 new outcomes added (all remaining STATUS_IN_PROGRESS players correctly skipped). **108 clean outcomes remain**, score range 60–75, 81.5% accuracy. The 276 ungraded predictions will complete tomorrow at 8 AM when Round 4 finishes.
